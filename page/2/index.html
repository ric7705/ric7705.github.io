python design pattern

發表於 2021-06-06 分類於 ceiere_hide

學完沒用到等於白學, 每種pattern學完用一句話來總結使用時機
要注意因為強型別語言和弱型別的差異, 所以實作上的差異可能也影響到概念

  • python: 弱型別, 如果要按照JAVA那套來做, 反而code會很不pythonic
  • JAVA: 強型別, 其實這種表達性不好的容易產生冗余code, 這是最需要使用DP的

Factory pattern

工廠內通常會有一些選擇條件, 如果要精簡也可以把if/else改為使用dict

結論

  • 概念:
    • 根據不同的參數給不同的物件
    • 在主區塊創物件的流程, 隱藏到工廠去做
  • 使用時機:
    • 拿物件時期, 如果物件有相似性(例如方法一樣), 可以考慮使用工廠統一負責產出
  • 一句話總結
    • 給string, 給你相對應的物件

Abstract Factory pattern

結論

  • 概念:
    • 一個抽象工廠(只是interface)去掌管各個子工廠
  • 使用時機:
    • 子項目繁多, 一層的factory pattern無法完成時, 可以考慮把工廠給抽象化

note: python的抽象工廠概念上比較沒有強型別語言的強, 硬要創介面code又不pythonic

Builder

結論

  • 概念:
    • 專注在一個物件的建立過程, 一步一步build起來的情況
    • 搭配Fluent interface, 看起來就是用很多.去建立起一個物件
  • 使用時機
    • 生產流程一樣, 但最終產品卻會不一樣時
  • cmp factory
    • 都是產出物件
    • factory建構的最小單位是一個物件
    • builder建構的最小單位可以是屬性
  • 一句話總結:
    • 一樣的建構流程, 但最終產品卻不一樣

note:
python跟JAVA的怎麼差那麼多…先跳過

decorator

注意, decorator這邊是design pattern
不是很多程式語法上的decorator(@)

結論

  • 概念:
    • 成品透過裝飾, 又形成另一種成品
    • 搭配巢狀function, 看起來就是用()去裝飾成另一個物件
      • new_item = step2_obj(step1_obj(old_item))
  • cmp bulder:
    • builder: 傾向於物件的組裝, 一開始沒有成品
    • decorator: 一開始有個成品, 透過裝飾形成另一個成品

Borge

也稱為Monostate pattern, 類似的作法為singleton

這個要直接看code觀察行為了

結論

  • 概念:
    • class及延伸的所有obj共享namespace
    • 修改dict pointer, 讓所有物件和這個類別共享相同mem區塊
  • 使用時機
    • 物件之間要sync狀態時
  • 一句話總結:
    • 所有物件和類別共享狀態

Proxy

結論

  • 概念:
    • 不直接去存取目標物件, 而是去請求代理幫你問
    • 有保護目標物件的概念
  • 使用時機:
    • 目標物件需保護時
    • 目標物件耗費效能時
    • 目標物件高延遲
    • 目標物件多人共享時
  • 一句話總結:
    • 請proxy幫你存取物件

Facade

“這個有可能是很多人不知不覺都會用到的design pattern”
不要和cascade(連鎖效應)搞混

結論

  • 概念:
    • 提供一介面讓client去存取某個子系統
  • 使用時機:
    • 如果不同類別有相同方法, 例如test, run等
  • 一句話總結:
    • 隱藏子系統(強型別)

Composite

“這個有可能是很多人不知不覺都會用到的design pattern”

結論

  • 概念:
    • 物件還有物件
    • 物件的屬性還是物件
  • 使用時機:
    • any time, 想把程式寫的結構化時
    • 要注意不要過度設計
  • 一句話總結
    • 不需要特別知道的design pattern, 除非新手

python exception

發表於 2021-06-06 分類於 ceiere_hide

概念

一但發生例外, 通常報完錯程式就立即停止了,
藉由自己捕捉例外, 當程式錯誤時可以自我進行修復或重試

架構

架構為:

  • BaseException
    • Exception
    • KeyboardInterrupt
    • SystemExit

最容易遇到的是Exception裡面的

  • Exception
    • ArithmeticError
    • RuntimeError
    • TypeError

比較幾種常見用法
- except:
- 捕抓所有的例外(從BaseException往下)
- except Exception as e:
- 只捕捉Exception的(從Exception往下)
- 使用as e可以順便查看錯訊
- except BaseException as e:
- 綜合前面兩者, 兼具查看錯訊和捕捉所有的功能

用法

finally: 通常用在必須關閉資源才離開程式時, 例如f.close()

搭配loop

若是遇到程式必須執行, 遇錯須重試時, 可搭配loo使用

拋出異常

使用raise, 有時候並不想立即處理錯誤, 而是往上拋給caller, 此時可使用raise
通常會在except區塊使用raise

raise後面的obj 或class可省略,代表直接回傳當前的例外

使用raise有點像傳炸彈
A -> B -> C -> D
如果D發生錯誤, 想要在B處理, 則
D捕抓到往上拋, C捕抓到往上拋, B捕抓到進行處理

raise 的層級需要再研究一下, 以及使用raise的時機

自定義例外

開發程式時, 有時候會想把例外包裝起來, 自己做控制flow
實現方法只要寫個繼承可能發生錯誤的類就好了(例如Exception, BaseException等)

情況複雜時, 亦可以結構化整個自定義的例外繼承規則, 例如

class Error(Exception):
class InputError(Error):
class APIError(Error):
等等

python mutable/imutable

發表於 2021-06-06 分類於 ceiere_hide

物件本身有三個元素:

  • identity
  • type:
  • value

identity

物件被建立後identity不會變, 可以想成是物件被放在mem的address

可以用is來比對物件的identity, 即比較address
可以用id來獲得物件的identity的整數表示

type

跟id一樣無法改變, 可使用type來獲取obj的型態

value

相對於is比較mem address, 比較value可使用==
有些obj在建立後value是可變得稱為mutable, 若否則為immutable

mutable: list, dictionary, set and user-defined classes
immutable: int, float, bool, str, tuple, unicode

對immutable改值後, 原本的值’依然存在mem’, 只是新分配一個mem出來而已
有這個概念比較好理解

list, tuple:

list是mutable, 所以id不會變, 但值改變了
tuple是immutable, 值改變後, id也變了(重新alloc一個mem給他)

int, string:

int, string很常用, 但其實是immutable, 對他改值一樣會重新alloc新的mem

使用immutable的index時有點tricky的地方:

你以為可執行, 只是a的id改變, 其實是會噴錯訊,
說tuple沒有assignment operatrion
這點要注意一下, 其實就注意tuple即可,
因為immutable又有container性質的只有tuple而已

immutable的assign

對immutable改值會觸發alloc mem, 那assign相同的值呢?
答案是看是否有container性質, 幸運的是同時具備contain+immutable的只有tuple

對immutable沒container性質的obj來說, mem不變

對tuple而言, 雖然內容看起來一樣, 但mem也變了

passing

mutable/immutable在passing上有不同情況發生

mutable

mutable的obj, pass的話會是call by reference

list是mutable, assign給其他變數, 或是傳遞給function,
實際上都是把變數指向相同mem,
所以以上會印出

immutable

印出:

0

str是immutable, 傳給其他var時, 此時是指向相同mem,
但一但其中有人改變value後, 會重新alloc一個mem給他
所以帶有點lazy的概念
這邊要記住, 誰改值的誰就會改變mem

container

有些type具有container性質, 對於可否變的探討,
只要知道container是存id即可

討論immutable container包含mutable obj的情況

例如:

先從value來探討, tuple不可變, 但包含了可變的list
對list改值沒問題

那id呢, list value改變id不變,
那tuple呢, 感覺內部值也改變了, 但其實不變,
對於container, 可以想像tuple是類似存這樣的東西:
(129392130, skills_mem_address)
雖然skills_mdm_address的value變了, 但因為id沒變, 所以對於tuple來說,
他不算改變, 所以id一樣沒變
(這東西會牽涉到copy(), deepcopy()議題)

討論immutable container包含immutable obj的情況
例如:

info是tuple, 包含三個元素, immutable int, immutable int, immutable container

這三個都是immutable, 又因為對immuatble改值後, 舊有值依然存在mem,
且最外層info obj依然指向舊的mem, 所以info的value, id依然不變

效率上的考量

  • immutable obj的存取速度較快
  • 但若對immutable改值會較慢, 因為必須alloc新的mem
  • 對於會改值的操作使用mutable較有效率

有的話改為全部使用absolute import

attempted relative import beyond top-level package

python的資料夾有分一般資料夾(folder)和套件(package),
普通的folder要被視為package要滿足兩個條件:

  • 必須包含init檔案
  • 不能是執行入口

這邊接下來要說什麼是top-level, 如何定義,
會踩到這個錯誤代表你以為的top-level跟python interpretor認知的不同

在main.py檔案執行import時,
可能會是from package1.sub_package import module1

這時候對於module1來說, 他的name是package.sub_package.module1,
name的第一個是啥, 就是該module的top level!!!
因為top level是package, 所以只要你relative是在package下都可正常運行

如果relative使用在package之外, 那就會吃這個錯誤, 因為已經超過該module的top level
